# Создание обзорной статьи

Запусти полный цикл создания обзорной статьи. Этот тип отличается от обычных статей: у нас нет готового исходника, фактуру нужно собрать самостоятельно.

Тема статьи: $ARGUMENTS

## Что важно понимать про этот процесс

Главный инструмент качества — **твоя проверка в основном контексте**, не агенты. Раньше в цепочке были агенты `plan-checker`, `text-checker`, `final-reviewer` — они вылизывали текст по чек-листам и делали его механическим, не видя полной картины. Теперь эти этапы делаешь ты: читаешь и правишь сам. Ты видишь весь текст целиком, помнишь историю разговора с пользователем, фактуру из research.md и примеры — поэтому именно ты ловишь блоки-наполнители, абстрактную статистику и нерелевантные аудитории детали.

Перед тем как писать — прочитай 1–2 статьи из `examples/essays/`, если они есть. Стиль формируется из примеров, а не из правил. Если примеров нет, работай по ред-политике и профилю проекта.

## Рабочая папка

Определи имя рабочей папки из темы:
1. Возьми тему из $ARGUMENTS
2. Сделай из нее короткое имя папки (транслит или ключевые слова, без пробелов, через дефис)
3. Рабочая папка: output/[имя папки]/
4. Создай папку командой: mkdir -p output/[имя папки]/

Все агенты сохраняют файлы в эту рабочую папку. Передавай путь к ней каждому агенту.

## Параметры для всех агентов
- Тип контента: overview
- Скилл с ред-политикой: redpolicy-overview (файл skills/redpolicy-overview/SKILL.md)
- Правила письма: skills/good-writing/SKILL.md и skills/good-writing/antipatterns.md
- Папка с примерами: examples/essays/

## Шаг 1: Исследование и сбор фактуры
Используй подагента **researcher**.
Передай ему: тему статьи ($ARGUMENTS), рабочую папку, профиль проекта или путь к нему, если он заполнен.

Дополнительные инструкции для researcher:
- Помимо сбора фактуры, найди 2-3 лучшие статьи по теме в интернете. Зафиксируй их структуру в разделе «Референсные статьи и их структура» в research.md. Для каждой статьи запиши: URL, структуру блоков, что удачно, что можно улучшить
- Это НЕ статьи для копирования. Это референсы для планировщика, чтобы он видел, как другие авторы подходят к теме, и мог сделать лучше

Результат: файл research.md в рабочей папке с полной фактурой по теме и анализом референсных статей.

## Шаг 2: Планирование
Используй подагента **planner**.
Передай ему: путь к файлу research.md в рабочей папке (это исходный материал), тип контента (overview), путь к ред-политике, рабочую папку.

Важно: для этого типа контента исходником является research.md, а не внешний файл.

Дополнительные инструкции для planner:
- Перед построением структуры прочитай раздел «Референсные статьи и их структура» в research.md. Проанализируй, как лучшие авторы раскрывают тему. Используй их подход как отправную точку, но не копируй. Подумай, что можно улучшить, добавить или убрать, чтобы наш текст был полезнее для читателя
- Для каждого блока указывать, какие факты из research.md НЕ включать и почему (нерелевантны аудитории, абстрактная статистика, детали для другой аудитории)
- Один заголовок = один вопрос читателя. Если получается два вопроса — разделять на два блока
- Проверять каждый блок: «Можно ли его убрать без потери практической пользы для читателя?»
- Проверять каждый факт: «Что читатель сделает с этой информацией?»

## Шаг 3: Проверка плана (ты в основном контексте)

**Агент plan-checker больше не используется.** Проверяешь план сам.

1. Прочитай [рабочая папка]/plan.md целиком, держа рядом research.md.
2. Прочитай 1–2 статьи из examples/essays/, если еще не читал.
3. Оцени план:
   - Каждый заголовок отвечает на один вопрос читателя? Два вопроса — два блока.
   - К каждому блоку: «можно ли его убрать без потери практической пользы?» Блоки-наполнители (тренды, прогнозы, абстрактная статистика без действия) вычеркивай.
   - К каждому факту: «что читатель сделает с этой информацией?» Абстрактные факты заменяй конкретными советами.
   - Нет ли в плане инструментов и деталей для другой аудитории?
   - Перед блоками про деньги, тарифы, доступ или ресурсы заложен контекст: кто платит, за что, когда это актуально читателю?
   - Важные факты из research.md не потеряны?
4. При необходимости переписывай план сам — не запускай обратно агента. Ты работаешь быстрее и не ломаешь собранную фактуру.

## Шаг 4: Согласование плана

Покажи пользователю [рабочая папка]/plan.md и расскажи коротко, что поменял после своей проверки. Спроси, одобряет ли план.

- Если одобряет — переходи к следующему шагу
- Если просит изменения — внеси правки в plan.md и покажи обновленную версию
- НЕ НАЧИНАЙ писать текст, пока план не будет явно одобрен словами типа «ок», «давай», «пиши», «одобряю»

## Шаг 5: Написание по блокам

Прочитай утвержденный план из рабочей папки (plan.md) и найди все пронумерованные блоки.
Разбей на пары: [1–2], [3–4], [5–6] и т.д. Если блоков нечетное число — последняя группа из одного блока.

Для каждой пары последовательно запускай подагента **writer**. Каждый следующий запускается только после завершения предыдущего.

Каждый агент **writer** получает:
1. Путь к research.md в рабочей папке (исходный материал)
2. Тип контента: overview
3. Ред-политика: skills/redpolicy-overview/SKILL.md
4. Правила письма: skills/good-writing/SKILL.md и skills/good-writing/antipatterns.md
5. **Эталон стиля в промпте:** если в `examples/essays/` есть примеры, 2–3 абзаца из статьи прямо встроены в промпт как образец вайба. Не «прочитай файл», а готовый текст в промпте — иначе агент не проникнется стилем.
6. Рабочую папку
7. Полный план: [рабочая папка]/plan.md (чтобы агент понимал траекторию всего текста)
8. Уже написанный текст: содержимое [рабочая папка]/draft.md (пустое для первой пары; нарастает с каждой итерацией)
9. Задание: «напиши блоки X и Y, добавь в конец [рабочая папка]/draft.md»

Дополнительные инструкции для каждого агента writer:
- Не начинать блоки с переходных фраз («Вот как это выглядит на практике», «Разброс цен огромен», «Чтобы понять, во что это выливается...»). Заголовок дает контекст — начинать сразу с сути.
- Перед блоками про деньги, тарифы, доступ или ресурсы — объяснять контекст: кто платит, за что, когда это актуально для читателя
- Абстрактные факты заменять конкретными советами: не «разница 50%», а «общайтесь на английском, если платите за токены»
- Не включать инструменты и детали для другой аудитории
- Не использовать букву с двумя точками; antipatterns.md — не чек-лист, а инструмент для сомнительных мест

После написания всех пар draft.md содержит полный черновик.

## Шаг 6: Финальная проверка (ты в основном контексте)

**Агенты text-checker и final-reviewer больше не используются.** Проверяешь и финализируешь сам.

1. Прочитай [рабочая папка]/draft.md целиком, держа рядом research.md.
2. Веди проверку по слоям:
   - **Факты против research.md.** Цифры, цены, названия и утверждения совпадают с собранной фактурой; ничего не додумано сверх нее.
   - **Структура.** Не дублирует ли подзаголовок лид (если да — убрать подзаголовок)? Каждый заголовок содержит только один вопрос?
   - **Польза.** Каждый факт: «что читатель сделает с этим?» Ничего — убирать или заменять на совет. Каждый инструмент: «может ли читатель из профиля проекта использовать это?» Нет — убирать. Нет ли блоков-наполнителей (тренды, прогнозы, абстрактная статистика без действия)?
   - **Контекст аудитории.** Перед блоками с ценами, доступом или ресурсами есть объяснение: кто платит, за что, когда актуально?
   - **Без буквы с двумя точками** по всему тексту.
   - **Книжные конструкции и нейромаркеры** по antipatterns.md — для сомнительных мест, не как протокол.
   - **Дубли** между блоками; выводы-обобщения, повторяющие сказанное.
3. Сделай два отдельных целевых прохода — по одному паттерну за раз:
   - **Переходные фразы.** Найди и убери «Вот как это выглядит», «Разброс огромен», «Чтобы понять, во что это выливается» и подобные — блок начинается сразу с сути.
   - **Чтение вслух.** Каждый абзац: «коллеге за кофе так бы объяснил?» Книжное — переписать целиком, не подправлять слово.
4. Вноси правки напрямую через Edit. Не делегируй агенту — он сломает стиль вылизыванием.
5. Скопируй итог в [рабочая папка]/final.md.

## Шаг 7: Сдача

Покажи пользователю [рабочая папка]/final.md и краткий отчет о правках, которые ты внес на шаге 6. Спроси, устраивает ли результат или нужны доработки.

Шаги 3 и 6 — это **твоя работа в основном контексте**, а не вызов агента. Не делегируй их обратно агентам `plan-checker`/`text-checker`/`final-reviewer` — архитектура осознанно построена так, что ты делаешь эти этапы сам.
