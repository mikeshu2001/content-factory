# Создание статьи-подборки сервисов

Запусти полный цикл создания статьи-подборки.

Исходный материал: $ARGUMENTS

## Что важно понимать про этот процесс

Главный инструмент качества — **твоя проверка в основном контексте**, не агенты. Раньше в цепочке были агенты `plan-checker`, `text-checker`, `final-reviewer` — они вылизывали текст по чек-листам и делали его механическим, не видя полной картины. Теперь эти этапы делаешь ты: читаешь и правишь сам. Ты видишь весь текст целиком, помнишь историю разговора с пользователем и знаешь примеры — поэтому именно ты ловишь книжные конструкции, дубли, расхождения с данными исследования.

Перед тем как писать — прочитай 1–2 подборки из `examples/articles/`, если они есть. Стиль формируется из примеров, а не из правил. Если примеров нет, работай по ред-политике и профилю проекта.

## Рабочая папка

Определи имя рабочей папки из имени исходного файла:
1. Возьми имя файла из $ARGUMENTS (без пути к директории и без расширения .md)
2. Рабочая папка: output/[имя файла]/
3. Создай папку командой: mkdir -p output/[имя файла]/

Все агенты сохраняют файлы в эту рабочую папку. Передавай путь к ней каждому агенту.

## Параметры для всех агентов
- Тип контента: article
- Скилл с ред-политикой: redpolicy-article (файл skills/redpolicy-article/SKILL.md)
- Правила письма: skills/good-writing/SKILL.md и skills/good-writing/antipatterns.md
- Папка с примерами: examples/articles/

## Шаг 1: Исследование сервисов
Используй подагента **article-researcher**.
Передай ему: путь к исходнику ($ARGUMENTS), рабочую папку.

Результат: файл research.md в рабочей папке с актуальными данными о каждом сервисе.

## Шаг 2: Планирование
Используй подагента **planner**.
Передай ему: путь к исходнику ($ARGUMENTS), путь к research.md в рабочей папке (дополнительный исходный материал с актуальными данными о сервисах), тип контента (article), путь к ред-политике, рабочую папку, путь к папке с примерами (прочитать 1–2 примера перед составлением плана).

Важно: planner использует ОБА файла — оригинальный исходник и research.md. Данные из research.md актуальнее.

## Шаг 3: Проверка плана (ты в основном контексте)

**Агент plan-checker больше не используется.** Проверяешь план сам.

1. Прочитай [рабочая папка]/plan.md целиком, держа рядом research.md.
2. Прочитай 1–2 подборки из examples/articles/, если еще не читал, — чтобы помнить целевую структуру.
3. Оцени план по ред-политике redpolicy-article:
   - Каждый сервис заложен по шаблону: название, ссылка, стоимость, описание?
   - Данные в плане совпадают с research.md (он актуальнее исходника)? Цены и статусы не взяты из устаревшего исходника?
   - Подборка логично упорядочена, нет сервисов-дублей с одинаковой ролью без объяснения разницы?
   - К каждому факту — вопрос «зачем это читателю?». Перегруз деталями тарифов и ограничений вычеркивай (см. antipatterns.md, раздел «Избыточные детали в описании сервисов»).
4. При необходимости переписывай план сам — не запускай обратно агента. Ты работаешь быстрее и не ломаешь собранную фактуру.

## Шаг 4: Согласование плана

Покажи пользователю [рабочая папка]/plan.md и расскажи коротко, что поменял после своей проверки. Спроси, одобряет ли план.

- Если одобряет — переходи к следующему шагу
- Если просит изменения — внеси правки в plan.md и покажи обновленную версию
- НЕ НАЧИНАЙ писать текст, пока план не будет явно одобрен словами типа «ок», «давай», «пиши», «одобряю»

## Шаг 5: Написание по блокам

Прочитай утвержденный план из рабочей папки (plan.md) и найди все пронумерованные блоки.
Разбей на пары: [1–2], [3–4], [5–6] и т.д. Если блоков нечетное число — последняя группа из одного блока.

Для каждой пары последовательно запускай подагента **writer**. Каждый следующий запускается только после завершения предыдущего.

Каждый агент **writer** получает:
1. Путь к исходнику ($ARGUMENTS)
2. Путь к research.md в рабочей папке (дополнительный исходный материал с актуальными данными о сервисах)
3. Тип контента: article
4. Ред-политика: skills/redpolicy-article/SKILL.md
5. Правила письма: skills/good-writing/SKILL.md и skills/good-writing/antipatterns.md
6. **Эталон стиля в промпте:** если в `examples/articles/` есть примеры, 2–3 описания сервисов прямо встроены в промпт как образец вайба. Не «прочитай файл», а готовый текст в промпте — иначе агент не проникнется стилем.
7. Рабочую папку
8. Полный план: [рабочая папка]/plan.md (чтобы агент понимал траекторию всего текста)
9. Уже написанный текст: содержимое [рабочая папка]/draft.md (пустое для первой пары; нарастает с каждой итерацией)
10. Задание: «напиши блоки X и Y, добавь в конец [рабочая папка]/draft.md»

Важно: writer использует ОБА файла — оригинальный исходник и research.md. Данные из research.md актуальнее.

Обязательные требования к каждому writer: шаблон описания сервиса по ред-политике; ссылки на каждый сервис; не использовать букву с двумя точками; antipatterns.md — не чек-лист, а инструмент для сомнительных мест.

После написания всех пар draft.md содержит полный черновик.

## Шаг 6: Финальная проверка (ты в основном контексте)

**Агенты text-checker и final-reviewer больше не используются.** Проверяешь и финализируешь сам.

1. Прочитай [рабочая папка]/draft.md целиком, держа рядом research.md.
2. Веди проверку по слоям:
   - **Факты против research.md.** Цены, тарифы, статусы и возможности каждого сервиса совпадают с исследованием. Расхождение — правишь по research.md.
   - **Шаблон сервиса.** Каждый сервис: название, ссылка, стоимость, описание. Ссылки на месте и вшиты в название, а не голым доменом.
   - **Соответствие ред-политике** redpolicy-article: структура подборки, лид, порядок сервисов.
   - **Без буквы с двумя точками** по всему тексту.
   - **Книжные конструкции и нейромаркеры** по antipatterns.md — особенно разделы «Избыточные детали в описании сервисов», «Перегруженные цены», «Маркетинговый тон».
   - **Дубли.** Стоимость не повторяется в описании; одинаковые формулировки не кочуют из сервиса в сервис.
   - **Сравнение с примерами.** Открой 1–2 подборки из examples/articles/ и сверь подачу.
3. Сделай два отдельных целевых прохода — по одному паттерну за раз:
   - **Охота на балласт.** По каждому предложению: «поможет ли это читателю выбрать сервис?» Нет — удаляй.
   - **Чтение вслух.** Каждое описание: «так бы посоветовал сервис коллеге?» Каталожное и рекламное — переписать целиком.
4. Вноси правки напрямую через Edit. Не делегируй агенту — он сломает стиль вылизыванием.
5. Скопируй итог в [рабочая папка]/final.md.

## Шаг 7: Сдача

Покажи пользователю [рабочая папка]/final.md и краткий отчет о правках, которые ты внес на шаге 6. Спроси, устраивает ли результат или нужны доработки.

## Выполняй шаги последовательно

Каждый следующий шаг начинается только после полного завершения предыдущего.

Шаги 3 и 6 — это **твоя работа в основном контексте**, а не вызов агента. Не делегируй их обратно агентам `plan-checker`/`text-checker`/`final-reviewer` — архитектура осознанно построена так, что ты делаешь эти этапы сам.
